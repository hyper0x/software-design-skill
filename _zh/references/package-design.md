# 包/模块设计原则

> 出处：Robert C. Martin，《Agile Software Development, Principles, Patterns, and Practices》（2002）

将类组织成包（或模块/目录）并管理它们之间依赖关系的原则。

对于单体应用，以下三条最重要：

---

## ADP — 无环依赖原则 (Acyclic Dependencies Principle)

> **包的依赖图不能有环。**

### 意图

当包 A 依赖包 B，包 B 又依赖包 A（直接或间接传递）时，就产生了环。环会导致：
- 无法理解依赖方向
- 无法独立测试包
- 无法推理变更影响

### 如何识别违规

- 循环 import 导致运行时错误
- 编译顺序问题（在编译型语言中）
- "我无法提取这段代码，因为它依赖所有东西"
- 一个包的变更需要重新构建所有东西

### 打破环的技巧

| 方法 | 怎么做 |
|------|--------|
| **依赖反转** | 让两个包都依赖第三个包中的接口 |
| **提取** | 把共享依赖移到自己的包中 |
| **合并** | 如果环很紧，它们应该是一个包 |

### 常见踩坑

- ADP 违规导致级联重编译——一个变更迫使所有东西重新编译
- 在 Go 中，环在编译时被捕获（好）。在 Python 中，它们导致运行时 import 错误。两种情况下都是设计坏味道
- 两个包之间的环几乎总是意味着它们应该是一个包，或者共享代码需要被提取出来

---

## CCP — 共同闭包原则 (Common Closure Principle)

> **一起变更的类放在一起。**

### 意图

如果两个类总是因为同一个理由被修改，它们应该在同一个包中。这最小化了受一次变更影响的包的数量。

### 如何应用

- 按**变更理由**分组，而不是按技术类别
- 不要把所有的"控制器"放在一个包、"服务"放在另一个包——按功能/领域分组

### 示例

**坏（按技术层分组）：**
```
payment/
├── controllers/
│   ├── OrderController.py
│   └── RefundController.py
├── services/
│   ├── OrderService.py
│   └── RefundService.py
├── models/
│   ├── Order.py
│   └── Refund.py
```

**好（按变更理由分组）：**
```
payment/
├── order/
│   ├── OrderController.py
│   ├── OrderService.py
│   └── Order.py
└── refund/
    ├── RefundController.py
    ├── RefundService.py
    └── Refund.py
```

### 常见踩坑

- CCP 是包级别的 SRP——一个包应该只有一个变更理由
- CCP 和 CRP（共同复用原则）存在张力——CCP 说"一起变更的放一起"，CRP 说"别让客户端依赖它们不用的东西"。在单体中，CCP 通常更重要

---

## SDP — 稳定依赖原则 (Stable Dependencies Principle)

> **朝稳定的方向依赖。**

### 意图

稳定的包（很少变化的东西）应该被依赖。不稳定的包（经常变化的东西）应该依赖稳定的包，而不是反过来。

### 什么使包"稳定"

一个包如果有很多入向依赖（很多东西依赖它）和很少出向依赖（它依赖很少东西），它就是稳定的。改变一个稳定包的代价很高，因为依赖它的所有东西可能都需要改变。

### 如何应用

```
业务逻辑（不稳定，经常变化）
    ↓ 依赖
工具库（稳定，很少变化）
```

而不是：
```
工具库（稳定）
    ↓ 依赖
业务逻辑（不稳定）  ← 坏：稳定依赖不稳定
```

### 常见踩坑

- SDP 不是说"稳定的包更好"——是说"不稳定的包应该依赖稳定的包"
- 抽象接口是最稳定的东西——它们没有需要变更的实现。这就是 DIP 有效的原因：两者都依赖抽象
- 实践中：工具包（字符串、日期、日志）是稳定的。业务逻辑包是不稳定的。业务逻辑应该依赖工具包，而不是反过来
