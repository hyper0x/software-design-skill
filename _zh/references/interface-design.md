# 接口设计原则

设计方法签名、API 边界和组件间契约的原则。

---

## 得墨忒耳定律 (Law of Demeter, LoD)

> 出处：美国东北大学，1987年。正式表述："每个单元对其他单元的了解应该有限。"

**只跟你的直接朋友说话。**

### 意图

一个方法只应调用以下对象的方法：
1. 自身
2. 自己的参数
3. 自己创建的对象
4. 自己的直接组件对象

它不应该**穿透**一个对象去调用第二个对象内部的方法。

### 如何识别违规

```python
# 违规：穿透对象
order.getCustomer().getAddress().getCity()

# 更好：告诉 order 你需要什么
order.getCustomerCity()
```

### 常见踩坑

- 得墨忒耳不是说"永远别链式调用"——是"别穿透对象去拿它的内部"
- 得墨忒耳适用于**内部结构**，不适用于流式 API 或建造者模式（`.setX().setY().build()`）——那些就是设计来链式调用的
- 违反得墨忒耳的真正代价：暴露了内部结构，使得无法在不破坏调用方的情况下修改

## 命令-查询分离 (Command-Query Separation, CQS)

> 出处：Bertrand Meyer，《面向对象软件构造》（1988）

**方法要么改状态，要么返回数据，不要同时做。**

### 意图

- **命令**：改变状态但不返回值的方法
- **查询**：返回数据但没有副作用的方法

当一个方法同时做这两件事时，调用方无法判断调用它会改变什么还是仅仅查询什么。

### 如何识别违规

```python
# 违规：既修改状态又返回数据
def pop(self) -> Item:
    self.count -= 1
    return self.items[self.count]

# 更好：分离命令和查询
def remove_last(self):
    self.count -= 1

def peek(self) -> Item:
    return self.items[self.count - 1]
```

### 常见踩坑

- CQS 不是说"修改方法不能返回值"——是"不要在查询方法里隐藏修改"
- 实践中，有些操作天生就是两者兼具（如 `pop()`）。这个原则是指导方针，不是绝对规则
- 最危险的违规：查询方法中缓存结果作为副作用——调用方不期望 `getX()` 会修改任何东西

## 鲁棒性原则 (Postel's Law / Robustness Principle)

> 出处：Jon Postel，RFC 761（TCP 规范），1980年

**发送保守，接收宽容。**

### 意图

你的代码应该产生格式良好、严格的输出。但在接收输入时，对小的变化要宽容。这让系统更健壮。

### 如何应用

- **发送**：始终产生正确、格式良好的输出。不要依赖接收方的宽容
- **接收**：接受合理的变化（空白、大小写、额外字段），但拒绝真正无效的输入

### 常见踩坑

- 鲁棒性原则说的是**输入宽容**，不是"什么都接受"——在系统边界严格校验
- 在安全敏感的上下文中，接收也要严格——宽容可能被利用
- 这个原则在现代协议设计中受到批评——边界的严格校验通常更受青睐

## 最小惊讶原则 (Principle of Least Surprise)

> 出处：通用 UX 原则，应用于软件设计

**函数的行为应与它的名字和签名一致。**

### 意图

代码应该是可预测的。当开发者读到 `user.save()` 时，他们应该能预测它的行为而不需要读实现。惊讶（副作用、意外行为）会侵蚀信任。

### 如何识别违规

```python
# 违规：意外的副作用
def save(self):
    self.db.persist(self)
    self.send_confirmation_email()  # 惊讶！调用方没想到

# 更好：明确的命名
def save(self):
    self.db.persist(self)

def save_and_notify(self):
    self.save()
    self.send_confirmation_email()
```

### 常见踩坑

- 最小惊讶违规通常来自**副作用**——`save()` 不应该发邮件
- 这个原则也适用于**错误行为**——如果一个函数可能失败，调用方应该能预测怎么失败
- 当你无法避免惊讶时，在函数名或文档字符串中明确说明
